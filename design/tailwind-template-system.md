Below is an updated **POC spec** for a Tailwind-templated theming system that:

* keeps your **existing UI authored in Tailwind utilities**
* generates a **stable “template stylesheet” at build/transpile time**
* supports **(1) existing default design**, **(2) curated preset themes**, and **(3) user+LLM generated themes**
* enables **instant theme switching** by applying **CSS variable overrides** (no full refresh required)

This spec is written to be applied to an **existing React + Node** codebase with minimal disruption, while making the theming system a first-class primitive.

---

# Tailwind-Templated Theming POC Spec

## 1) Goals

### Primary goals

1. **Tailwind remains the authoring surface**
   Components keep `className="..."` utility styling as the main mechanism.

2. Build produces a **template stylesheet** that “just works” with injected themes
   The CSS bundle references **theme variables** (CSS custom properties), so changing variables changes the look without regenerating Tailwind.

3. Support three theme sources:

* **Default theme = your existing design** (baseline)
* **Curated themes** (ship with product)
* **User-generated themes** (LLM → JSON → validated → CSS variable overrides)

4. **Realtime switching**

* Preset theme switches are instant (attribute toggle).
* User theme apply is instant after the LLM returns (single style tag swap).

### Non-goals (POC)

* Public theme marketplace, moderation, permissions
* Arbitrary user CSS (we only allow variable assignments)
* Perfect coverage of every edge in the existing codebase on day one

---

## 2) Tailwind v4 requirement (recommended for the POC)

This spec assumes **Tailwind CSS v4** because it gives you the cleanest “template CSS” story:

* You define design tokens as **theme variables** using `@theme` in CSS, which determine what utilities exist and also generate regular CSS variables you can override at runtime. ([Tailwind CSS][1])
* Core namespaces exist for the primitives we want to theme: `--color-*`, `--font-*`, `--spacing-*`, `--radius-*`, `--shadow-*`, etc. ([Tailwind CSS][1])
* Spacing utilities like `p-*` are driven by a single `--spacing` theme variable, making “density” and sizing genuinely themeable. ([Tailwind CSS][2])
* Tailwind supports arbitrary values and CSS variables, and even includes a shorthand for padding like `p-(--my-padding)` (equivalent to `p-[var(--my-padding)]`). ([Tailwind CSS][2])

If you’re on v3 today, the system still works conceptually, but “global density” becomes harder. For the POC, upgrading to v4 is the simplest path.

---

## 3) Key idea: Template CSS + Overrides

### Template CSS (transpile/build time)

* One compiled CSS bundle generated by Tailwind.
* This bundle defines utilities that ultimately resolve through CSS variables.

### Theme overrides (runtime or shipped)

* **Overrides never add new utilities.**
* Overrides only set CSS custom properties (variables) under a theme selector.
* Switching themes becomes: “change which variables are active.”

---

## 4) Core primitives

### 4.1 Theme scope primitive

Use a single attribute as the “theme selector”:

* Put theme state on `html` (recommended for existing apps, covers portals/modals):

```html
<html data-theme="default">
```

* Also support `.theme-scope` for previews inside the Theme Lab page:

```css
:root, .theme-scope { /* defaults */ }
:root[data-theme="cyberpunk"], .theme-scope[data-theme="cyberpunk"] { /* overrides */ }
```

### 4.2 Theme types

* **Default theme**: the variables that reproduce the current design.
* **Preset theme**: curated variables shipped with the app.
* **User theme**: variables generated from JSON; injected via a runtime `<style>` tag.

### 4.3 “Runtime theme style tag”

For user themes, inject/replace exactly one style tag:

```html
<style id="runtime-theme"></style>
```

Applying a user theme is just:

* `runtimeThemeStyle.textContent = cssText`
* `document.documentElement.dataset.theme = "user:<id>"`

## 5) Build-time system requirements (Template generation)

### 5.1 Files to add

**A) `src/styles/app.css`** (the “abstraction file” + Tailwind import)

This is where your *existing design tokens* are encoded in a Tailwind-native way.

```css
@import "tailwindcss";

/* Theme variables define which utilities exist and generate CSS variables. */
@theme {
  /* Semantic colors (NOT Tailwind palette colors). */
  --color-bg: #0b0b10;
  --color-surface: #141423;
  --color-surface-2: #1b1b2e;
  --color-fg: #f2f3ff;
  --color-muted: #b9bad6;
  --color-border: color-mix(in oklab, var(--color-fg) 14%, transparent);

  --color-accent: #a855f7;
  --color-accent-contrast: #0b0b10;

  /* Typography */
  --font-sans: ui-sans-serif, system-ui, sans-serif;
  --font-mono: ui-monospace, SFMono-Regular, Menlo, monospace;

  /* Density: Tailwind spacing utilities are derived from --spacing */
  --spacing: 0.25rem;

  /* Shape + shadows */
  --radius-md: 10px;
  --radius-lg: 14px;
  --shadow-sm: 0 1px 0 rgba(0,0,0,.18);
  --shadow-md: 0 6px 0 rgba(0,0,0,.22);
}

/* Non-Tailwind variables (layout knobs that don’t need new utilities) */
:root {
  --app-container-max: 1120px;
  --app-card-pad: 16px;
  --app-grid-cols: repeat(3, minmax(0, 1fr));
}
```

Notes:

* `@theme` is where Tailwind v4 wants you defining tokens. ([Tailwind CSS][1])
* Namespaces like `--color-*`, `--font-*`, `--radius-*`, `--shadow-*`, and spacing namespaces are how Tailwind maps tokens to utilities. ([Tailwind CSS][1])
* Spacing utilities driven by `--spacing` lets a theme change density without rewriting every class. ([Tailwind CSS][2])

**B) `src/styles/themes.presets.css`** (curated themes)
This file contains curated overrides only:

```css
:root[data-theme="default"],
.theme-scope[data-theme="default"] {
  /* optional: you can keep this empty if defaults are in :root */
}

:root[data-theme="cyberpunk"],
.theme-scope[data-theme="cyberpunk"] {
  --color-bg: #05060a;
  --color-surface: #0b1020;
  --color-fg: #d7faff;
  --color-accent: #00ffd5;

  --spacing: 0.28rem;
  --radius-lg: 6px;

  --app-grid-cols: repeat(2, minmax(0, 1fr));
}
```

**C) Build output**
Your bundler should produce:

* `app.css` (Tailwind compiled, stable template)
* `themes.presets.css` (preset overrides)

These can be concatenated into one CSS bundle if you prefer.

## 6) Authoring requirements for existing Tailwind components

This is the critical part: **your components must use semantic utilities** that resolve via variables, not hard-coded palette classes.

### 6.1 Allowed patterns (themeable)

✅ Semantic colors:

* `bg-bg`, `bg-surface`, `text-fg`, `text-muted`, `border-border`, `bg-accent`, `text-accent-contrast`

✅ Radius + shadow utilities that map to theme variables:

* `rounded-md`, `rounded-lg`
* `shadow-sm`, `shadow-md`

✅ Density-based spacing:

* `p-4`, `gap-3`, `mt-6` (driven by `--spacing`) ([Tailwind CSS][2])

✅ Layout knobs via CSS variables + arbitrary values:

* `max-w-[var(--app-container-max)]`
* `p-[var(--app-card-pad)]`
* `grid-cols-[var(--app-grid-cols)]`

(If you like the shorthand form for padding variables, Tailwind supports `p-(--my-padding)` as sugar. ([Tailwind CSS][2]))

### 6.2 Disallowed patterns (or “legacy only”)

🚫 Hard-coded palette classes in new code:

* `bg-white`, `text-gray-900`, `border-slate-200`, etc.

🚫 Runtime-generated class strings:

* `className={"bg-" + color}` (Tailwind won’t generate it reliably and it bypasses theming discipline)

### 6.3 Migration strategy for an existing codebase

You will not convert everything at once. The POC should prove the pipeline + primitives.

**Step 0: Tokenize without changing UI**

* Add the `@theme` defaults to match the existing look exactly.
* Add presets file (even if only 1 theme exists).

**Step 1: Convert shell + primitives**

* Migrate: app shell, nav, page background, cards, buttons, inputs to semantic tokens.
* These primitives give you immediate “coverage” across the product.

**Step 2: Convert the 3 test surfaces**

* Rating feed page
* Forum/thread page
* Shop/listing/checkout page

**Step 3: Add lint guardrails**

* Add an ESLint rule or custom lint that warns on raw palette classes in non-legacy directories.

## 7) Realtime theme switching primitives (Frontend)

### 7.1 ThemeManager module (required)

A small framework-free controller:

**Responsibilities**

* Apply preset theme by setting `data-theme`
* Apply user theme by injecting cssText + setting `data-theme`
* Persist selection
* Provide “reset”
* Provide subscription for React state updates

**Pseudo-API**

```ts
type ThemeSelection =
  | { kind: "preset"; id: string }
  | { kind: "user"; id: string; cssText: string };

interface ThemeManager {
  applyPreset(id: string): void;
  applyUserTheme(id: string, cssText: string): void;
  reset(): void;

  getSelection(): ThemeSelection | null;
  subscribe(fn: () => void): () => void;
}
```

### 7.2 Apply logic (atomic)

Preset:

1. `document.documentElement.dataset.theme = presetId`

User:

1. `#runtime-theme.textContent = cssText`
2. `document.documentElement.dataset.theme = "user:" + userThemeId`

This avoids partial application where theme is selected but variables aren’t ready.

### 7.3 No-flash bootstrapping (required)

Persist theme in localStorage and apply it **before React mounts**:

* inline tiny script in `<head>` that:

  * reads saved selection
  * sets `#runtime-theme` (if user theme)
  * sets `data-theme`

This prevents the “default theme flash” on reload.

## 8) Theme sources

### 8.1 Default theme (existing design)

* The `@theme { ... }` values in `app.css` represent your current design system.

### 8.2 Curated preset themes (ship with app)

* Stored as CSS override blocks in `themes.presets.css`.
* Bundled with app.
* Switching is instant and offline.

### 8.3 User themes (LLM-generated)

* Represented as a **ThemeSpec JSON** + compiled CSS variable overrides.
* Stored server-side (optional for POC), cached client-side.
* Applied via the runtime theme style tag.

## 9) Backend primitives (Node)

### 9.1 Endpoints

**`POST /api/themes/generate`**

* Input: `{ prompt, basePresetId? }`
* Output: `{ themeId, themeSpec, cssText, warnings[] }`

**`POST /api/themes/compile`** (dev tool)

* Input: `{ themeSpec }`
* Output: `{ cssText, warnings[] }`

**`GET /api/themes/presets`**

* Output: `{ presets: [{ id, name, tags }] }`

### 9.2 ThemeSpec constraints (security + robustness)

* LLM returns **JSON only** (no CSS).
* Server validates schema + clamps values.
* Server derives secondary tokens (hover, border strength, muted text) to keep JSON small and consistent.

### 9.3 CSS compilation rules (very important)

When compiling user theme overrides:

* Output **only**:

  * `:root[data-theme="user:<id>"] { --var: value; ... }`
  * and optionally `.theme-scope[data-theme="user:<id>"] { ... }`

* Allow only value types you can parse and re-serialize:

  * colors (hex / rgb / oklch)
  * lengths (px/rem)
  * numbers (unitless)
  * a very small set of “complex” vars you explicitly allow (like `--app-grid-cols` containing a grid template string)

Reject anything else (no `url()`, no `@import`, no arbitrary selectors).
